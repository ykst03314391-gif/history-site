# 戦国時代 情報サイト（仮）

日本の戦国時代（おおよそ 15 世紀末〜16 世紀末）を、**管理人が自分の視点で捉え直す個人メディア**。
**戦国時代が好きな人が「新しい情報を仕入れたい」ときに訪れる場**——新説・お出かけ情報（展示・イベント）・ゲーム情報をネットから定期・不定期に集めて記事化し、あわせて人物・合戦・史跡のデータベースも更新し続ける。
考察（記事）を主役に据え、DBはそれを支える「脚注」として構造化する。事実は出典に基づき、そのうえで「自分はこう考える」という態度を前面に出す。

## コンセプト（背骨）

**戦国時代を、自分の視点で捉え直す個人メディア。**

このサイトを貫く「串」は、テーマでもジャンルでもなく、**管理人という一人の視点**である。
歴史上の事実は出典に基づいて正確に扱う。そのうえで、「自分はこう捉える」という**態度**を前面に出すことが、このサイトの存在理由であり、最大の差別化。

事実だけなら Wikipedia がある。ここにしかないのは、**一人の人間が戦国時代をどう見ているか**という視点そのもの。

### だれが、なぜ訪れるのか

読者像は明確。**「戦国時代が好きで、新しい情報を仕入れたい人」**。
その人が「何か新しいネタはないかな」と思ったとき、まず覗きに来る——**戦国ファンの『情報の仕入れ先（アンテナ）』** になることを目指す。

だから、じっくり読む考察（A）だけでなく、**移り変わる最新情報（B）**も定期・不定期に追いかけて"お知らせ"として蓄積する。そのうえで、その情報を**ネタに管理人が考察（A）を書き起こす**——**お知らせで気づき、考察で深める**、という流れをつくる。

### 看板は「視点」、最新情報は「呼び込み」

AとBを対等の二本柱にはしない。**看板（サイトの魂）は常に A＝視点・考察**。最新情報（B）は、その視点を当てる"新しい素材"であり、来訪の入口にすぎない。並べて見せると、速報・網羅で勝る大手に埋もれ、差別化が薄れるからだ。

**A. 深める（考察・特集｜evergreen）＝価値の中心**
じっくり読み解く定番。「ここでしか読めない」をここに置く。**看板はいつもこちら。**
- 経済で見る戦国時代 ／ 社会の通説っておかしくね？ ／ 戦国ゆかりの地めぐり

**B. 仕入れる（最新情報｜入口・フック）＝お知らせ・鮮度**
新説・お出かけ・ゲームの新着を **「情報のお知らせ」にとどめる**（何が出た／いつ／どこ／出どころ）。ここでは踏み込んだ考察はしない——**考察は A の仕事**。B は定期・不定期に**自動的に蓄積**され、A を書くための"ネタ帳"にもなる。
- **新説・新研究**：話題の新説・新刊の告知（→ 深掘りは A「通説」特集で）
- **お出かけ情報**：博物館の特別展・戦国イベントなど会期もの（→ A「旅」特集や地域へ）
- **ゲーム系**：信長の野望など新作・更新の告知（→ 実像との比較は A の企画で）

> **すみわけ：B はお知らせ（軽い・蓄積型）、A はそれをネタに書く考察（労作・手動）。** 差別化できるのは A の視点だけで、B の"お知らせ"単体は大手（ニュース・公式）に埋もれる。だから B を看板にしない。
> B はネットから定期・不定期に収集して**蓄積**し、関連する DB（人物・合戦・史跡）も更新する。A は管理人が時間をかけて**書き上げる**もの。
> 一言でいえば **「B で気づいて、A で深める」**。看板は常に A。

### 一貫する態度：「別の角度から捉え直す」

通説をなぞらない。教科書の順番で並べない。
**「本当にそうか？」「別の見方はないか？」** と問い直し、自分なりの答えを出す——この態度が全コンテンツに共通する背骨。

特集はこの態度の「現れ」であって、それぞれ別テーマではない。同じ一人の視点を、違う角度から当てているだけ。

| 特集 | 当てている角度 |
|---|---|
| **経済で見る戦国時代**（economics） | 大名を「経営者」に見立て、知行・恩賞・収入という**カネの角度**から捉え直す |
| **社会の通説っておかしくね？**（myth-busting） | 広まった通説を、資料に基づいて**疑いの角度**から捉え直す |
| **戦国ゆかりの地めぐり**（旅・travel） | 机上の知識を、実際に立つ**現地の角度**から捉え直す |

> どの特集も、やっていることは同じ：**「別の角度から、自分の目で捉え直す」**。
> 新しい特集を足すときも、この態度に接続できるか（＝一人の視点の現れになっているか）を基準にする。

### 考察が主役、データベースは脚注

このサイトの主役は**考察（記事）**。人物・合戦・大名家のデータベースは、主役をやめて**考察を支える「脚注」** の役割に置く。

- 記事を読んでいて「この人物、どんな人だっけ？」となったとき、すぐ引ける**足場**としてDBがある。
- DBは「網羅」を目指さない。**考察が必要とする範囲**を、必要な深さで整える。
- 入口も「調べる」より「読む・捉え直す」を先に置く。

> 従来は「DB（主）＋記事（従）」で考えていたが、これを反転させる。
> **視点のない事実の集積は Wikipedia の劣化版**にしかならない。主役は常に「管理人の視点」。

### このサイトのスタンス（明示することが差別化）

差別化の核は、機能でも網羅性でもなく、**スタンスをはっきり掲げること**。

- 事実は出典に基づく。諸説あるものは諸説あると書く。
- そのうえで、**「自分はこう考える」を隠さない**。中立を装わない。
- 通説・定説・ゲームの評価であっても、資料に照らして**再評価する**（例：「信長の野望、実は間違ってる？」）。
- 読者を「調べて終わり」にせず、**読む本・行く現地**という次の行動へつなぐ。

> 常に自問する：これは「事実の再掲」に留まっていないか。**自分の視点は、どこに出ているか。**

## 運用方針

- **出典重視**：掲載情報は可能な限り出典を明記する。諸説ある場合はその旨を記載
- **裏取りはネット中心で十分**：Wikisource（一次史料の翻刻）・NDLデジタル（活字本OCR）・J-STAGE／機関リポジトリ（オープンアクセス論文）・CiNii（書誌）で当たれる範囲を検証する。**網羅（完全性）は目指さない**——当たれない史料は「諸説あり／未確認」と正直に書く。**追跡可能性と正直さこそが質**（大半の個人ブログは出典なし＝ここが差別化）。詳細な当たり先は [`docs/sources.md`](docs/sources.md)。
- **継続更新**：新説・新研究に応じて内容をブラッシュアップしていく

> ⚠️ **現状のDBデータは全て「仮置き」**：人物・合戦・大名家の各データは、いまのところ管理人（および生成）の一般知識で書いた暫定値で、**記載中の出典（信長公記など）は未検証のラベルにすぎない**。一次資料での裏取りは未実施。DB系ページ（`person`/`people`/`battle`/`battles`/`clan`/`clans`/`regions`）には「仮置きデータ」注記を表示している。順次、原典に当たって修正し、検証済みになった記述から `sourceId` を正しく紐付けていく。

### 編集ルール：大名家の「一門」の範囲

大名家ページで「一門」と「ゆかりの人物（家臣／その他）」を分類する際の基準。
判断軸は「**その家に入るか／家から出るか**」。

**一門に含める**
- 当主の血族（父母・兄弟姉妹・子・孫・甥姪 など）
- 家に入った近親：正室、迎えた養子、婿養子（娘婿）
- 他家へ養子に出た実子（血がつながるため。「◯◯家を継ぐ」と注記する）

**一門に含めない**
- 他家へ嫁いだ実娘（婚姻で嫁ぎ先の家人となる＝姻戚。その人物ページの家族欄には残すが、実家の大名家ページの一門には載せない）
- 血縁も姻戚もない家臣（→「家臣」に分類）

> ポイント：娘婿・養子は「家に入る」ため一門。実娘を嫁がせるのは「家から出る」ため一門としない。
> 詳細は [`docs/schema-clan.md`](docs/schema-clan.md) を参照。

## マネタイズ（構想）

### 優先順位（大前提）

このサイトは **「自分がやりたいこと・自分にとって有益」が第一**。マネタイズは二の次。お金に引っ張られて提灯記事を書くと、唯一の差別化（正直な視点）が死ぬ（→ [`docs/competitors.md`](docs/competitors.md)）。

- **free-first**：まず無料で読める考察・トピックを積む。読者の信頼と作品群が育つのが先。
- **仕組みは置くが、押し付けない**：アフィリの土台（`references`/`buy`）は用意済みだが、これは「次に読む本」への**読者サービス**であって収益ノルマではない。仮タグのまま眠らせ、埋めるのは気が向いた時でよい。
- **課金は当面しない**：有料 note・メンバーシップは**読者から需要が出てから**。自分で「価値がある」と決めて先に課金しない（成功例も皆、信頼が育った後に収益化）。まずは低摩擦の**投げ銭**があれば十分。

基本の手段は**アフィリエイト**。コンテンツの「次の行動」（読む本・行く現地・遊ぶゲーム）に自然につなぐ形で、内容ごとに提携先を使い分ける。

### 提携先の方針

- **書籍・ゲーム** → **もしもアフィリエイト**（Amazon・楽天を一括で扱え、Amazon審査も通りやすい）＋**楽天アフィリエイト**（審査ゆるく即開始）
- **お出かけ・旅** → **楽天トラベル／じゃらん**（宿泊）、**アソビュー**（博物館・体験チケット）
- **横断（幅広い案件）** → **A8.net**、将来的に **Google AdSense**（クリック/表示型広告）

### 掲載ルール（可読性・誠実さと両立させる）

- **考察・記事（A）**：本文に広告・ボタンを挟まない。**記事末尾の「関連書籍」だけ**に留める（＝引用した本への導線）。
- **トピック（B）**：入口・フックなので、詳細ページに**購入・予約ボタンを出してよい**（お出かけ＝宿・チケット、ゲーム＝購入、新説＝関連書籍）。
- **開示は必須**：アフィリンクの近くに開示文を表示（Amazon利用時は「Amazonのアソシエイトとして…適格販売により収入を得ています」を明記）。`rel="sponsored nofollow noopener"` を付ける。
- 文脈に関係するものだけを載せる。ランダムな商品広告は置かない。

> 実装済み：記事の `references`（関連書籍）とトピックの `buy`（多店舗対応）。開示文とラベルは掲載先に応じて自動で切り替わる。詳細は [`docs/schema-topic.md`](docs/schema-topic.md)。

### アフィリエイト以外の収益手段（候補）

相性順（このサイトに合う順）に。どれも「PR・広告・支援」を明示し、**考察の中立性は売らない**（提灯記事にしない）ことが前提。

1. **読者から直接（最も背骨に合う）** ― 「一人の視点」に対価を払ってもらう形。
   - **投げ銭**（OFUSE／codoc／Buy Me a Coffee）を記事末に。
   - **note の有料記事／有料マガジン**で深い考察を課金（無料で引き、深掘りを有料）。
   - **メンバーシップ**（pixivFANBOX／Fantia／Ci-en）で限定考察・早期公開・史跡写真。
2. **ふるさと納税アフィリエイト**（楽天ふるさと納税・さとふる） ― 城・史跡のある自治体へ。「地域から探す／お出かけ」と直結。
3. **ディスプレイ広告**（Google AdSense） ― 受動収入。ただし**少量**に留め可読性を優先。
4. **自作物を売る** ― 電子書籍/PDF（「城めぐりガイド」等、Kindle・BOOTH・note）、**オリジナルキャラ「ぱぱねこ」グッズ**（SUZURI等のPOD）・LINEスタンプ。
5. **旅行体験・動画配信・物販の拡張** ― KKday/Klook（城ツアー・甲冑体験）、U-NEXT等（大河・歴史番組）、御城印・レプリカ等。
6. **タイアップ・スポンサー（アクセスが育ってから）** ― 博物館・観光協会・自治体のPR（お出かけ）、出版社・ゲーム会社の新刊/新作PR（新説・ゲーム）、純広告。
7. **その他** ― クイズの広告/一部課金（企画済み）、実績が付けば寄稿・監修・講演。

> 当面：無料コンテンツを積むのが最優先。収益は**アフィリの仕組みを眠らせて維持**＋（やるなら）低摩擦の**投げ銭**まで。**有料 note・メンバーシップ・AdSense は当面見送り**（需要・アクセスが育ってから検討）。

### 他サイトとの共存（リンク方針）

網羅は専業に任せ、**リンクで送客して共存**する（我々は「視点と考察」担当）。
- 城データ → **攻城団**、展示網羅 → **インターネットミュージアム** 等へリンク。
- **リンクは自由**（公開ページへのハイパーリンクは許可不要）。
- **転載はしない**：相手の文章・写真・データのコピーは要許可＝やらない。ロゴ・画像の無断使用もしない。

### 記事の置き場所（note 等との棲み分け）

考察記事の「家」は**本サイト**。note 等の外部プラットフォームに記事を**引っ越さない**。
- 理由：考察は看板であり、記事⇄人物⇄合戦⇄地図⇄トピックの**横断統合（＝差別化）は自サイト内でしか効かない**。所有権・回遊・SEO の観点でも自サイトが正典。
- note を使うなら**逆向き**に：note には**抜粋・導入だけ**を置き、末尾で本サイトへ誘導する（**note → 本サイト**の集客）。全文の二重掲載は避け、canonical は本サイト。

### 地図（暫定：OpenStreetMap → 公開時に Google マップへ）

合戦の「古戦場（現在地）」やお出かけトピックの会場地図は、`place.lat`/`lng` から地図を埋め込んでいる。

- **現在（暫定）**：**OpenStreetMap 埋め込み**（APIキー不要）。座標から即生成でき、開発中は無料・手間なし。Google マップへは「開く」リンクで補完。
- **最終形（決定）**：**Google マップの埋め込み（Maps Embed API）に切り替える**。座標から `…/maps/embed/v1/place?key=<KEY>&q=<lat>,<lng>` で表示。
- **切り替えのタイミング**：サイトの作成が一通り終わり、**公開の目途が立った段階**で実施する（それまでは OSM のまま進める）。
- **切り替え時の作業**：Google Cloud で **Maps Embed API キー**を発行 → **HTTP リファラ制限**をかける → 地図生成箇所（`js/battle.js`・`js/topic.js` の OSM iframe）を Google Embed に差し替える。

## ローカルでの起動方法

データを `fetch` で読み込むため、**HTML ファイルを直接ダブルクリック（`file://`）で開くと動作しません**。
簡易 HTTP サーバー経由で表示してください。

プロジェクトのルート（`history-site/`）で、以下のいずれかを実行します。

```bash
# Python がある場合
python -m http.server 8765

# Node.js がある場合
npx serve -l 8765
# もしくは
npx http-server -p 8765
```
